    推送流程¶

    ssh密匙制作¶

    生成/添加SSH公钥 - Gitee.com

    要将项目开源到平台，首要问题是本地客户端与远端服务器的网络通信。常用通信协议是ssh，这需要配置公钥密钥，作为连接远端服务器的令牌。制作方式见Gitee。

    制作完的秘钥可应用应用在各种网络通信环节，只要这把秘钥与部署在服务器的公钥匹配。故推荐存在User目录下吧。

    hub上，可以图形化地为账号和仓库配置公匙，在设置里找找就行

    初始化本地仓库并推送到远程端¶

    本地仓库初始化：¶

    git init    %初始化本地仓库 cmd中，% 是注释符
git branch  -M [master] lctee/lchub %branch：分支命令 [-M]参数： 将[]分支重命名
git add *   %添加工作区所有文件的"变化增量"到暂存区中
git commit -m [message]    %提交暂存区的所有修改，作为仓库的一个版本记录

    推送到到远程仓库¶

    若远程仓库为空，为根据['master']参数创建初始分支

    git remote add [gitee/github] http/ssh.git  %新建远程仓库
git push -u lchub github 'master'  %推送[本地分支]到[远程仓库]的[远程分支]，并设置好此本地分支的默认远程分支 -u:updata

    多分支推送多仓库¶

    git branch lctee  %创建新的分支
git checkout lctee %切换分支

%设置此分支连接远程分支
git branch --track %启动分支追踪模式 
%<远程主机/远程分支>
git branch --set-upstream-to gitee/master  %建立本地分支到远程分支的持久关联

%拉取远程分支，并生成分支合并(远程与本地)工作流，
git pull gitee [master] : [lctee]   %设定好当前分支和追踪分支后，后两个参数可以不填
%审阅代码冲突
%合并完成后，可以向远程push了
%<远程主机名> <本地commit>[:<远程分支名>] 远程分支同名或已设置追踪时，可省略。
git push -u gitee lctee[:'master']  
%-u参数：推送并执行--set-upstream-to，建立本地分支到远程分支的持久关联

    其中，push内容为本次的commit。

    分支拉取与合并冲突 git fetch/pull¶

    拉取分支最近一次修改事件

    git fetch [github master] 仅拉取文件，不执行合并工作流

    git merge 合并，略

    git pull [github master] 拉取并合并，随后是审阅代码冲突

    代码冲突审阅流程：git status粗略对比文件增量差异；

    git pull 后，拉取到了一个新的commit 版本并将其与本地工作区文件合并，冲突部分会被修改称带冲突标记<<<<<<<、=======和>>>>>>>的样式，用户可用冲突审阅工具或记事重新修改文件，完成冲突合并操作。

    此后使用 git add将增量变化添加到暂存区。

    GIT工作模型 工作区 暂存区 版本库¶

    工作区

    本地仓库的实际项目文件。

    暂存区

    GIT 的暂存区(stage)是一个文件，它记录从上一个commit提交开始，工作区文件的增量快照,其结构为：文件列表索引index和文件的增量快照cache。

    版本库

    GIT 的版本库是一个数据库，它存储了项目的所有历史记录和版本信息。 git commit 命令将暂存区增量快照提交到版本库，形成一个基于文件commit的所有增量快照。但对其的 add,rm等操作，均是在这个基础快照上进行的。

    或者使用 git checkout 命令从版本库中检出文件到工作区。

    工作流程

    GIT 的工作流程是这样的：您在工作区中修改或创建文件，然后使用 git add 命令将文件的增量快照（即相对于上一次提交的修改部分）添加到暂存区，最后使用 git commit 命令将暂存区中的所有增量快照转换为一个永久的版本记录，并保存到版本库中。

    版本控制命令手册¶

    首先要摒弃git是为远程仓库服务的挂念。git的一切命令基于本地的分布式版本控制需求。

    git add/rm/commit¶

    git add 为添加工作区文件的增量快照（即文件的改动部分）到暂存区中。

    git rm将文件的暂存区增量和工作区均删除；

    git rm --cached不记录这部分变化,（距上一个commit的工作区变更）。仅删除残存区变化。

    git add使暂存区的增量记录与工作区的文件迭代同步；git rm作用于被git add的文件增量时，暂存区的增量记录弱后工作区，状态和对工作区修改后，未git add时等价。

    git commit用于提交暂存区内记录的所有快照，做为一次版本记录为版本库的HEAD；包含了当前工作区的增量，父提交的引用，以及一些元数据，如作者，日期，和消息。

    git commit的常见参数有以下几个：

    • -m <msg>指定提交消息（显示在hub的文件名侧边）
    • -a 自动 git add * ,再commit
    • -C/c <commit> 复用某次提交的附带信息。区别为是否再启动编辑器修改日志信息
    • git commit -m <msg> 为本次提交附带日志信息。

    git tag¶

    git tag是一个git命令，用于给仓库中的某个提交打上标签，以示重要或者标记一个版本。git tag支持两种类型的标签：轻量级标签（lightweight）和附注标签（annotated）。轻量级标签只是一个指向特定提交的引用，不包含任何额外的信息。附注标签是一个独立的对象，包含了标签名、消息、作者、日期等信息，还可以用GPG签名和验证。

    • git tag的常用参数有以下几个：
    • -l 或 --list：列出已有的标签，可以带正则模板来匹配特定的标签。如git tag -l "v1.0*"会列出所有以v1.0开头的标签。
    • -a 或 --annotate：创建一个附注标签，需要提供一个标签名和一个消息。如果没有指定消息，git会启动编辑器让你输入。例如，git tag -a v1.0 -m "Version 1.0"会创建一个名为v1.0的附注标签，并附上消息"Version 1.0"。
    • -m 或 --message：指定创建附注标签时的消息，可以和-a选项一起使用。例如，git tag -a v1.0 -m "Version 1.0"。
    • -d 或 --delete：删除一个或多个已有的本地标签，需要提供一个或多个标签名。例如，git tag -d v1.0会删除v1.0这个本地标签。

    git restore¶

    git restore用以从一个恢复原回复工作区指定文件，将其回滚到指定的commit或index。

    可以使用--source或者-s选项来指定恢复源1，默认为最近一次的commit，暨HEAD指针

    git restore ：将丢弃工作区的未保存修改(未 git add),回滚到指定的恢复原。（包括对文件的增删修改）

    • git restore的常见参数有以下几个：
    • <filename>：指定要恢复的文件名（或多个文件名）。您可以提供单个文件名/路径，也可以提供多个文件名（用空格分隔），或者使用通配符模式（例如*.html）。另一种选择是提供.字符，从而恢复当前目录中的所有文件。
    • –staged：将文件从暂存区中移除，但保留其实际修改。默认情况下，git restore命令会丢弃相应文件中的任何本地未提交的更改，并恢复其最后提交的状态。使用–staged选项时，文件只会从暂存区中移除，但其实际修改不会受到影响。
    • –source <src>：恢复文件的特定修订版本。默认情况下，文件将恢复到其最后提交的状态（或者只是取消暂存）。–source选项允许您在特定修订版本处恢复文件。
    • –patch：允许您选择要恢复的单个块。Git以交互方式逐步执行更改中的所有单独块，并询问您是否要丢弃/取消暂存每个块。请注意，–patch可以接受没有路径规范，并将提示恢复所有修改过的路径。

    git checkout¶

    “签出”语法，有两种常见复用

    • git checkout <branchName>切换分支;不带具体的文件路径时，用于修改head当前指向的分支

    • git checkout <fileName>：签出文件，放弃暂存区add的所有修改增量，从HEAD commit 恢复index，然后从index恢复工作区。作用对象为针对文件的回滚

    • git tag的常用参数有以下几个：

    • -b 创建分支并切换。git branch <new> ;git checkout <new>

    总之，就是让这个工作区文件回滚到最近一次git commit或git add时的状态。

    git reset¶

    操作对象：head指针。修改改head指向commit，回滚到以往的commit。同时根据修改head后，是否级联修改stage、workspace，可进一步的细分成三种模式hard、sort、mixed（默认），如下图所示：

    • commit：是指respoistory中每次保存（提交）后的结果，指向一个完整的版本记录
    • head：是一个指针，代表当前正在操纵的版本（commit），可以指向feature、master等有名分支，也只指向某一commit id 直接代表的版本

    git reset --hard [github/master] 参数为commit标识符，可以是远端仓库。

    • [当您使用git reset --soft时，您的索引不会改变，也就是说，您之前暂存的更改仍然保留在索引中。您的工作目录也不会改变，也就是说，您之前修改的文件仍然保留在工作目录中。这样，如果您再次执行git commit命令，您会得到一个和之前相同的提交]

    • [当您使用git reset --mixed时（注意：这是默认选项），您的索引会改变，也就是说，您之前暂存的更改会被取消暂存，并恢复到指定提交版本的状态。您的工作目录不会改变，也就是说，您之前修改的文件仍然保留在工作目录中，但是它们会显示为未暂存的更改]

    • --hard

    会重置工作区，作用在文件上，用于项目的版本回滚。

    大文件限制¶

    解决方案如下脚本：自动搜索项目文件，将大于50M的文件写入某配置文件。（忽略 或 大文件托管）

    type nul >.gitignore
echo ".gitignore" >>.gitignore
echo "/large-file-ignore.sh" >>.gitignore
## 查找大于 50MB 的文件并将它们的相对路径写入 .gitattributes 文件中
find . -type f -not -path './.git/*' -size +50M -exec sh -c 'echo "/$(realpath --relative-to=. "{}") " >> .gitignore' \;
